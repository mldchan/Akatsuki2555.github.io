Akabot Privacy statement

About Privacy Policy Terms of Service Changelog Documentation

The public instance of Akabot Discord bot takes privacy very seriously. It is as private as if you would host it yourself. If you think this privacy statement is way too privacy intrusive, check out the Discord privacy policy.

What Akabot stores in its databases

Each feature of the bot stores as little data as possible. Here's a quick overview of what is stored and what is associated with every feature:

  • Analytics - There's a feature in Akabot which counts which command is executed how many times, globally. This feature doesn't associate any command runs or button clicks with any users or servers, it only counts them. This feature is here to see which features are used the most on the bot and what should be added or improved in the future.
  • **Anti Raid** - The Anti Raid feature stores only your settings, and if you count the RAM of the server as storage, it stores the number of joins and the number of messages and the dates they were last updated for a maximum of the user's desired time. No user ID's or messages are stored, only numbers.
  • **Automod Actions** - User defined actions along with all information about them (names, actions and the optional message) are stored. When an automod action with a name gets triggered, the user defined actions stored will be executed.
  • **Bot Help** - The bot help commands don't store any user information.
  • **Chat Revive** - The Chat Revive feature only stores which role has been chosen, channels which are active and the last message time sent in that channel.
  • **Chat Streaks** - The Chat Streaks feature stored the date you started your chat streak along with the date of your last sent message. Using these 2 dates, the bot calculates the rest of the messages. You can disable the notifications of this feature using `/user_settings chat_streaks_alerts off` and you can always check your streak by using the `/streak` command.
  • **Chat Summary** - The Chat Summary feature stores the amount of messages sent in a channel daily, along with who has sent how many messages in that channel. Only the top 5 chatters will be shown in the summary. No message content is stored. All numbers in this module are cleared every midnight.
  • **Feedback Commands** - This module does explicitly state before usage that it does share user entered information, along with the user name and user ID with GitHub. No user information is saved with these commands. If the user doesn't want this to happen, they can use GitHub with their personal accounts. Any of the other commands like `/about` , `/privacy` , `/tos` do not save any user information.
  • **Giveaways** - The giveaways module stores the channel the giveaway was started in, the item entered and the ending date, calculated when the giveaway is created. The creator of the giveaway is not stored.
  • **Leveling** - Each user has an associated number of XP points which is stored and associated to every user. A list of leveling multipliers, the server general leveling multiplier, and the XP required to level up are stored and associated to every server.
  • **Logging** - The logging channel is stored and associated with the server. The bot then listens to Discord changes and creates messages from available information in your logs channel.
  • **Moderation** - No information is stored when using kick, ban and timeout commands. When using warnings, a list of warnings created using the bot are stored. Warning actions are also stored and associated with the server and performed after adding a warning to the user.
  • **Per User Settings** - All per user settings have settings associated with the user.
  • **Reaction Roles** - You wouldn't believe it, but this module doesn't store any information at all. The Role ID's and types of the reaction roles are stored in the button's custom ID itself.
  • **Roles on Join** - The list of roles to be added on join are stored and associated with the server.
  • **Server Settings** - All settings stored are associated with the server.
  • **Verification** - The verification difficulty and role are stored per server.
  • **Welcome and Goodbye** - All user defined settings are stored and associated with the server.

Backend modules mostly don't mess with any user data or any server data.

  • **Heartbeat** - Sends periodic network requests to a URL every set interval.
  • **Power Outage Announcements** - Stores the current time every minute, doesn't associate it with any server or user. Reads the last saved time and current system time on startup of the bot and makes an announcement in a channel preconfigured in the bot configuration if the bot was offline for a longer period of time.

The developer doesn't have any access to read user data or any user control over the bot. They always write scripts to update the database and performs it on the test database, which contains one server and 5 users, 4 of them are all variants of the bot. The only exception is analytics and the number of servers.

Linked or connected services

The services that are connected to the bot are listed below:

  • **GlitchTip** is used to capture errors. Errors do not show me any personal information, only generic messages. These errors aren't associated with any server or user, only with the line of code which caused the error. The used GlitchTip instance is a self-hosted one, running on the same server as the bot itself.

Third parties

The Discord bot can share data with GitHub, only if a user executes one of the `/feedback bug` or `/feedback suggest` commands. These commands explicitly state before running that they share this data with GitHub.

The bot doesn't share any user data with any other party, except the one listed above.

Changes to the privacy statement

If the privacy statement ever changes, which it might in the future, a Discord ping in the support server will alert you about it, along with a summary of the changes, so you don't have to read the entire statement again. You can join the Discord server [here](/discord).
